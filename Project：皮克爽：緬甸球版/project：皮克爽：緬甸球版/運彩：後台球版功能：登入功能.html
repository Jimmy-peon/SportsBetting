登入頁面

運彩：後台球版功能

功能說明： 提供後台使用者登入與登出的基本功能。登入時需輸入帳號、密碼與隨機驗證碼（圖形驗證碼）以驗證身分。登入成功後依使用者層級導向相應後台頁面。登出則會回到登入頁面。

欄位說明

欄位名稱 說明
帳號輸入 使用者登入帳號，為系統中已註冊的有效帳號
密碼輸入 登入密碼，需與帳號對應，登入時將進行加密驗證
隨機驗證碼 圖形驗證碼（CAPTCHA），每次登入畫面載入時隨機產生
NG圖片的二進制位元資料轉成Base64格式
每次登入畫面產生新圖片，驗證碼有效時間為 3 分鐘

功能操作說明

功能操作 操作方式 系統行為說明
登入 輸入帳號、密碼與正確的隨機驗證碼後提交表單 驗證成功則登入系統，並導向對應權限的後台首頁；驗證失敗則顯示錯誤訊息
隨機驗證碼更新 點擊驗證碼圖片或「重新產生驗證碼」按鈕 即時產生新的驗證碼圖案，並更新顯示
登出 在後台介面中，點擊的「登出」按鈕 返回登入頁面

註：

  • 若重複登入（後踢前），為同一帳號於另一裝置或瀏覽器重新登入
    • 系統會自動登出原先登入中的使用者，並保留最新登入者帳號，顯示「帳號已於其他裝置登入」提示

使用之API

取得登入資訊

  • 進入「登入」頁面時使用

登入

  • StatusCode＝3，彈窗文字內容為「驗證碼已過期，請重新輸入一次」，並更新畫面中的驗證碼圖片

刷新驗證碼

登出

  • StatusCode＝1，將使用者返回至「登入」頁面

說明

  • 客戶端識別碼不存在(已過期)時，跳出彈窗內容為「超過閒置逾時時間，請重新登入」，並再次「取得登入資訊」
  • 上述各 API 未提及狀態碼，彈窗文字內容會為「狀態訊息」內容

註解

目前使用的技術

captcha_id + server 端快取（如 Redis / memory）

  • 後端產生驗證碼時，給一組隨機 captcha_id
  • 將正確答案以 captcha_id 為 key 存入 Redis 或 memory cache
  • 前端每次提交時，附上 captcha_id 與 user_input
  • 後端查 Redis 比對答案，驗證後刪除該 key（一次性），就能避免機器人不斷使用相同的 captcha_id 登入後台

範例：

POST /api/get-captcha 
→ 後端回傳：{ "captcha_id": "abc123", "image": "base64..." }  

POST /api/submit-form
 {  
	"captcha_id": "abc123",
	"captcha_input": "7Xkwr" 
	"username": "john",
	"password": "MySecretPwd123"
 }

`

Redis 中可能儲存：
captcha:abc123 => 7Xkwr (EX 5m)

captcha_id 本身只是 ID，但在整體驗證碼架構中，它支援以下防機制：

防禦方式 captcha_id 的角色
一人一題驗證 每次生成驗證碼就產生唯一 captcha_id，防止一個答案被共用
無法預測 captcha_id 使用亂數或 UUID，不易猜測
不可重用 驗證一次後立即刪除 captcha_id，防止重送
可配合風控策略 系統可記錄特定 captcha_id 重複錯誤次數、IP 來源等，加入風險分析
無需使用 session 避免機器人利用 cookie/session 穿透

安全性功能

  • 白名單

    • 說明：設定固定IP才能進入系統
    • 常用情境：資料庫驗證
    • 缺點：浮動IP，不適合使用（wi-fi、手機網路）

  • 登入時的隨機驗證碼

    • 說明：防止機器人不斷登入此系統（DDos）
